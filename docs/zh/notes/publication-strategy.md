# 发布与开源笔记

这份文档适合放到 GitHub 开源，但需要把“公开可见”和“多人合写”分开看。技术书和代码不同，代码有接口、测试和客观对错，很多人提 PR 可以收敛；书更依赖单一作者声音。更现实的模式是：作者维护主线，社区贡献勘误、例子、术语讨论、翻译和 issue，而不是让多人重写章节。

GitHub 的价值主要是可见度、反馈和可信度。它可以作为完整内容沉淀，也可以作为简历和面试里的作品。README 需要诚实说明定位：这不是声称 AIOS、MemGPT 或 Compound AI Systems 不存在，而是一份带分布式系统和生产可靠性视角的综述和工程地图。

贡献边界应该提前写清楚：欢迎 typo、事实修正、参考资料、例子和翻译；主线结构、章节顺序和核心观点由作者维护。文字类项目可以使用 Creative Commons 许可证，而不是代码项目常用的 MIT 许可证。

## X 的用法

X 更适合作为分发入口，而不是承载完整章节。一个 thread 只讲一个点，不要试图把一章压进一条帖子。书是素材库，X 是单点传播。

最适合作为第一条 thread 的点是“蒸馏并不直接减少 token”。这个点有反共识、有成本模型、有工程取舍，能独立成立。

## Thread 草稿：蒸馏省的是哪一项成本

1. 看到有人说他靠蒸馏来省 token。方向其实错位了。蒸馏几乎不减少 token，它主要砍的是另一项成本。

2. 一次 agent 调用的成本可以拆成：调用次数 × 单次 token 量 × 单 token 成本。三个因子，对应三套完全不同的优化手段。

3. 小模型和大模型吃进去的可能是同样多的 token。蒸馏换的是每个 token 的计算单价，不是 token 数量本身。用分布式语言说，它是把任务下沉到更便宜的计算层，不是减少 RPC 或缩小 payload。

4. 真正减少 token 量的是另一组手段：RAG、上下文裁剪、摘要、Prompt Caching、压缩工具输出，以及给 Planner 设迭代上限。这些通常零训练，改工程结构和配置就能见效。

5. 唯一能让蒸馏顺带减少 token 的情况，是它把“多轮大模型规划”塌缩成“一次小模型处理”。这时省的是被省掉的那几轮调用，token 下降是副作用。

6. 如果目标是省钱，顺序通常是：先榨干缓存、裁剪和调用次数，再上路由或小模型，最后才考虑蒸馏。蒸馏需要标注、训练、评估和托管，是稳定流量和数据之后才值得上的牌。

不需要等整本书完成再公开。第一部分已经成型，再补一两个有差异化的章节，比如生产可靠性或安全，就可以开始收反馈。X 上可以每周拆一个点，GitHub 负责沉淀完整内容。
